# copilot-openai-proxy 代码审查结论（2026-04-23）

## 一、简短结论

当前版本**整体合理，可继续作为主线版本使用**。

核心原因：
1. 已切到**零自动重放优先**，方向正确
2. 对“**最小化高级请求**”目标更友好
3. stale tool repair / missing tool result / timeout retry 三条高风险链路都已明显收敛

结论级判断：
- **现在不用大改**
- **优先修统计口径，不优先改主恢复策略**
- **Claude native proxy 暂不并入当前代理**

---

## 二、已合理项

### 1. 恢复策略合理
- stale synthetic repair：本地忽略，不自动 replay
- missing tool result：本地阻断，不自动补跑
- silent-retry：仅限 `didReachCopilot !== true`

### 2. 会话状态回收基本完整
- `awaitingUserInput`
- `activeTurn`
- `pendingToolCalls`
- session disconnect / map 清理

### 3. 整体目标一致
当前代码和你的目标是一致的：
- 少额外请求
- 少高级请求浪费
- 少脏状态延续
- 错误更可解释

---

## 三、需要优化的点（按优先级排序）

### P1. usage / token 统计口径还不够稳
#### 现象
- live usage 与 shutdown usage 混用
- 某些轮次可能先显示 0，后面才补上
- token 可能存在“显示不稳定”而非“真实错误”

#### 风险
- 影响计费/观察判断
- 会让你误判某次请求是否真的消耗了 token

#### 可实现方法
1. 明确分成两类口径：
   - `live_estimated_usage`
   - `final_shutdown_usage`
2. `/metrics` 与响应体里增加 `usageFinal: boolean`
3. 只有 shutdown usage 到位后，才进入“最终累计统计”
4. liveUsage 只用于展示，不直接混入长期累计值

#### 建议
**先做。** 这是最值得补的一项。

---

### P2. metrics 里 timeout / completed 口径混杂
#### 现象
当前一次请求可能出现：
- 先记一次 timeout
- retry 后又记一次 completed

#### 风险
- `/metrics` 里的 timeout 偏高
- 用户看到“明明成功了，为什么 timeout 也涨了”

#### 可实现方法
把指标拆成两层：

1. **attempt 级**
   - upstream attempt started
   - timeout attempts
   - error attempts

2. **request 级**
   - user-visible completed
   - user-visible failed

#### 建议
**第二优先级。** 不影响主功能，但影响后续监控判断。

---

### P3. ask_user 强注入策略偏重
#### 现象
默认 `COPILOT_FORCE_ASK_USER=1`，并且 system prompt 强制：
- 临结束优先 ask_user
- 原则上不主动结束本轮

#### 风险
- 增加 `awaiting_user_input` 状态占比
- 拉长会话生命周期
- 某些本可直接结束的请求被挂起

#### 可实现方法
改成分层策略：

1. 默认关闭全局强制 ask_user
2. 仅在以下场景开启：
   - 当前渠道明确要求（如 webchat）
   - 当前 turn 使用过工具
   - 当前任务明显为多步骤持续任务
3. 增加 env 开关：
   - `COPILOT_FORCE_ASK_USER=0/1`
   - `COPILOT_FORCE_ASK_USER_WHEN_TOOLS_USED=1`

#### 建议
**第三优先级。** 不一定立刻改，但值得收紧。

---

### P4. `didReachCopilot` 仍是启发式判断
#### 现象
当前判断依赖：
- assistant message
- stream delta
- tool activity
- requestId

#### 风险
极少数情况下，可能已触达上游但尚未来得及产出事件，仍被误判为未触达。

#### 可实现方法
1. 如 SDK 可提供更早期 request ack / request accepted event，优先接入
2. 没有的话，保留当前启发式，但在日志里单独标注：
   - `didReachCopilot: inferred`
   - `didReachCopilotConfidence: high|medium|low`

#### 建议
**先不急改。** 当前版本已足够保守。

---

### P5. fallback 模块边界不清晰
#### 现象
`fallback.mjs` 存在，但当前主链路里不是特别清晰地启用。

#### 风险
- 后续维护者误以为已有稳定 fallback
- fallback 本身可能引入额外请求，不符合“最小高级请求”目标

#### 可实现方法
二选一：
1. 明确标注为实验模块
2. 或明确接线，并限定只在低风险场景触发

#### 建议
**先标注，不急接入。**

---

## 四、不建议现在动的部分

### 1. 不建议回退到旧恢复策略
原因：
- 旧策略更偏自动自愈
- 会增加隐性请求和高级请求消耗
- 脏状态更容易延续

### 2. 不建议现在把 Claude native 直通并进当前代理
原因：
- 当前代理已经在“最小高级请求”上表现较好
- 混入 Anthropic native path 会增加路由、错误、统计复杂度
- 更适合后续单独做并行代理

### 3. 不建议大改 session 主状态机
原因：
- 当前主逻辑已趋于稳定
- 真正问题更多在统计口径和策略过重，不在主状态机结构

---

## 五、推荐后续路线

### 最优先
1. 稳定 usage 最终口径
2. 拆分 metrics 的 request/attempt 统计

### 次优先
3. 收紧 ask_user 强制策略
4. 给 fallback 标清边界

### 暂缓
5. Claude native proxy 单独立项，不并入当前代理

---

## 六、最终结论

### 当前是否合理？
**是，合理。**

### 当前是否值得继续作为主线？
**是，值得。**

### 现在最该优化哪里？
1. **usage/token 统计口径**
2. **metrics timeout/completed 统计拆层**
3. **ask_user 注入策略收紧**

### 可实现方法是否明确？
**明确。** 上述三项都可以在现有架构上局部改进，无需推翻主链路。
